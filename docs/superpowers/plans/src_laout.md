Transitioning to a `src/` layout is a fantastic move. It is the gold standard for Python packaging because it prevents accidental import bleed and makes your app perfectly compatible with native installers like `uv tool` or `pipx`.

It can be a bit tedious to update all the imports, but you only have to do it once. Here is the step-by-step architectural plan to execute this transition cleanly.

### Phase 1: Directory Restructuring

First, we create the new namespace and move your application code inside it. The `tests/` directory and project config files stay at the root.

1. **Create the directories:**
   ```bash
   mkdir -p src/orion
   ```
2. **Make it a Python package:**
   Create an empty `__init__.py` file to mark `orion` as a package.
   ```bash
   touch src/orion/__init__.py
   ```
3. **Move your source files:**
   Move all your functional modules into `src/orion/`. 
   ```bash
   mv core memory safety tools ui src/orion/
   mv config.py src/orion/
   mv main.py src/orion/
   ```

**Your resulting tree should look like this:**
```text
cli-assistant/
├── pyproject.toml
├── uv.lock
├── README.md
├── project_overview.md
├── tests/                 # Stays at the root!
│   ├── conftest.py
│   └── test_*.py
└── src/
    └── orion/
        ├── __init__.py
        ├── main.py        # The core logic
        ├── config.py
        ├── core/
        ├── memory/
        ├── safety/
        ├── tools/
        └── ui/
```

### Phase 2: Updating Your Imports (The Tedious Part)

Since your code now lives inside the `orion` package, all your absolute imports need to be updated. 

You must change imports referencing your top-level folders to reference the `orion` namespace. You can use your IDE's "Find and Replace in Files" for this.

**Examples of what to change:**
* Change `from core.agent import build_agent` ➡️ `from orion.core.agent import build_agent`
* Change `from memory.db import ...` ➡️ `from orion.memory.db import ...`
* Change `import config` ➡️ `from orion import config`
* Change `from tools.files import ...` ➡️ `from orion.tools.files import ...`

*Tip:* Do not use relative imports (like `from ..core import agent`) across major subsystems. Stick to absolute package imports (`from orion.core import agent`); they are much easier to read and refactor later.

### Phase 3: Creating the Synchronous Entry Point

CLI entry points (the native binaries generated by Python installers) must point to a **synchronous** function. Since Orion uses `asyncio` for its core loop, you need a small sync wrapper in your `src/orion/main.py`.

Open `src/orion/main.py`. At the bottom of the file, you likely have something like this right now:
```python
if __name__ == "__main__":
    asyncio.run(main())
```

Change it to this:
```python
def cli_entry():
    """Synchronous entry point for the generated CLI tool."""
    import asyncio
    # Handle graceful exits for standard CLI interruptions
    try:
        asyncio.run(main())
    except KeyboardInterrupt:
        print("\nExiting Orion.")

if __name__ == "__main__":
    cli_entry()
```

### Phase 4: Updating `pyproject.toml`

Now you tell `uv` how to build your package and generate the CLI command. 

1. **Add a build system:** If you don't already have one, `hatchling` is the modern standard (and what `uv` defaults to).
2. **Define the project scripts:** Tell the installer to map the command `orion` to the `cli_entry` function you just made.

Add/modify these sections in your `pyproject.toml`:

```toml
[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

[project]
name = "orion-cli"
version = "0.1.0"
description = "A Linux-first CLI AI assistant"
readme = "README.md"
requires-python = ">=3.13"
dependencies = [
    "ddgs",
    "fastembed",
    "httpx",
    "playwright",
    "prompt-toolkit",
    "pydantic-ai",
    "rich",
    "sqlite-vec",
    "trafilatura",
]

[project.scripts]
# This creates the global `orion` command.
# Format: "command_name = module.path:function_name"
orion = "orion.main:cli_entry"
```

### Phase 5: Fixing the Tests

Your tests are still at the root in the `tests/` folder. They will fail until you update their imports to match the new `orion` namespace.

1. Do the same Find and Replace in the `tests/` directory: change `from core...` to `from orion.core...`.
2. Ensure your `pytest` environment knows where the source code is. When running tests with `uv`, `uv run pytest` usually figures this out automatically if you have the `src` layout.

### Phase 6: Implementing the Self-Healing Playwright Start

To make the app truly plug-and-play as a global tool, implement the auto-download for Playwright we discussed earlier. 

Inside `src/orion/ui/startup.py` (or wherever your initial boot sequence runs):

```python
import subprocess
import os
from rich.console import Console

console = Console()

def ensure_browser_engine():
    """Checks for Playwright binaries and installs them if missing."""
    cache_dir = os.path.expanduser("~/.cache/ms-playwright")
    if not os.path.exists(cache_dir):
        with console.status("[bold blue]Setting up web extraction engine for the first time...[/bold blue]"):
            try:
                subprocess.run(
                    ["playwright", "install", "webkit"], 
                    check=True, 
                    stdout=subprocess.DEVNULL, 
                    stderr=subprocess.DEVNULL
                )
            except subprocess.CalledProcessError:
                console.print("[bold red]Warning: Failed to install Playwright dependencies. Web extraction may not work.[/bold red]")

# Call this early in your startup sequence!
```

### Phase 7: Verification

Once you have made these changes, you can test if the architecture holds together locally before pushing it to GitHub.

1. From the root of your project, tell `uv` to install your local directory as a global tool:
   ```bash
   uv tool install .
   ```
2. Verify it is in your path:
   ```bash
   which orion
   ```
   *(It should point to something like `~/.local/bin/orion`)*
3. Run it:
   ```bash
   orion
   ```

If it boots up correctly, your architecture conversion is complete. You can now use the ultra-clean `install.sh` script we discussed previously! Let me know if you want to dial in any specific part of this.
